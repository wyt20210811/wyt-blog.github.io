组件

  • Fixture：一个或一组测试用例的测试对象。可以是要测试的对象或者函数。

  • TestCase：测试用例。是对测试对象的某个功能或流程编写的测试代码。对一个Fixture，可能有多个测试用例。通过继承它来实现测试用例。继承自Test和TestFixture，用户需要从TestCase派生出子类并实现runTest以开发自己所需的测试用例。

  • TestSuite：同时运行的测试用例的集合。可以是一个fixture的多个测试函数、也可以是多个fixture的所有测试用例。

  • TestFixture：测试固件。用来建立测试基准或构建测试用例的先决条件。用于维护一组测试用例的上下文环境。在实际应用中，经常会开发一组测试用例来对某个类的接口加以测试，而这些测试用例很可能具有相同的初始化和清理代码，所以引入TestFixture实现这一机制。它通常具有两个接口，分别用于测试环境的初始化与清理virtual void setUp() virtual void tearDown()

  • TestRunner：用来运行测试用例。接口为：virtual void addTest(Test *test)，virtual void run( TestResult &controller, const std::string &testPath = "" );TestRunner可以自行管理测试对象的生命周期，不用我们自己delete掉new的对象

  • TestListener：用来接收执行测试用例过程中产生的结果。

  • TestResult：用来管理TestListener。TestResult中的protect方法，作用是对执行函数的错误信息进行捕获，从而实现对测试结果的统计

  • TestCaller：TestCase适配器（Adapter），它将成员函数转换成测试用例。解决TestCase只能支持一个测试用例的问题，可以支持一组测试用例,如下

  • 使用时，在测试的主文件中将TestCase注册到TestSuite中，并运行，示例：
    Outputter：结果输出，可以定制不同输出格式
  • CppUnit定义一组宏用于检测错误，断言失败时，就会把错误信息报告给TestResult
  • TestFactory：测试工厂，这是一个辅助类，通过借助一系列宏定义让测试用例的组织管理变得自动化

简单测试 TEST

测试夹具 TEST_F()

  • 为多个不同的测试复用相同的对象配置。在使用TSET_F()之前，必须先定义一个测试夹具类（该夹具类继承自::testing::Test），并且需要重载默认构造函数SetUp()和析构函数。对于通过 TEST_F() 定义的每个测试，googletest 将在运行时创建一个全新的测试夹具，立即通过 SetUp() 初始化它，运行测试，通过调用 TearDown() 清理资源，然后删除测试夹具。注意相同测试套件中的不同测试夹具具有不同的测试夹具类，且 googletest 总是在创建下一个之前删除测试夹具。

  • 模拟接口的工具

定义Mock接口类

  • 使用MOCK_METHOD后加一个数字，表明需要mock的函数有几个参数，MOCK_METHOD宏的第一个参数是函数名，第二个参数是函数指针类型。

Gmock使用方法

  • EXPECT_CALL(mock_object, Method(argument-matchers))：声明一个调用期待，就是我们期待这个对象的这个方法按照什么逻辑去执行。mock_object是mock对象（如是TestUser的一个对象），Method是mock对象中的mock方法，其参数通过argument-matchers规则匹配
    .With(multi-argument-matchers)：多个参数的匹配方式指定
    .Times(cardinality)：这个方法可以被执行的次数，超过则按照默认值返回
    .InSequence(sequences)：指定函数执行顺序，通过同一序列中声明期待的顺序确定
    .After(expectations)：用于指定一个方法只能用在另一个方法以后
    .WillOnce(action)：执行一次方法时，将执行其参数action的方法
    .WillRepeatedly(action)：一直调用一个方法时，执行其参数action的方法
    .RetiresOnSaturation();保证期待调用不会被相同的函数的期待所覆盖
    where all clauses are optional, and .InSequence()/.After()/.WillOnce() can appear any number of times.

jmeter

  • 用于接口测试和压力测试，基于JAVA，使用前需要先安装jdk。安装： (16条消息) Jmeter的基本使用_M_Huayra的博客-CSDN博客_jmeter使用 。
  • 组件包含：①、 测试计划是使用 JMeter 进行测试的起点，它是其它 JMeter 测试元件的容器。②、线程组：代表一定数量的并发用户，它可以用来模拟并发用户发送请求。③、取样器（ Sampler ）：模拟各种请求。所有实际的测试任务都由取样器承担，存在很多种请求。 如：HTTP 、 ftp 请求等等。④、监听器：负责收集测试结果，同时也被告知了结果显示的方式。功能是对取样器的请求结果显示、统计一些数据（吞吐量、 KB/S…… ）等。⑤、断言：用于来判断请求响应的结果是否如用户所期望，是否正确，非常实用⑥、定时器：负责定义请求（线程）之间的延迟间隔，模拟对服务器的连续请求。⑦、逻辑控制器：允许自定义 JMeter 发送请求的行为逻辑，它与 Sampler 结合使用可以模拟复杂请求序列。⑧、 配置元件维护 Sampler 需要的配置信息，并根据实际的需要会修改请求的内容。⑨、 前置处理器和后置处理器负责在生成请求之前和之后完成工作。前置处理器常常用来修改请求的设置，后置处理器则常常用来处理响应的数据。

创建线程组

  • 线程组就是按照线程性质进行分组。一个进程=多个线程组，一个线程组=多个线程 
  • 设置线程数来模拟用户数量： 线程数：就是虚拟用户的数量，一个虚拟用户占用一个线程；准备时常：这些个虚拟用户要多长时间全部启动；循环次数：每个用户（线程）发送的请求次数；调度器：设置线程组启动的开始时间和结束时间；持续时间：测试持续的时间；启动延迟：测试延迟启动时间

sampler采样器

  • 向服务器发送请求，记录响应时间的最小单元和响应信息。如发送HTTP请求：名称：用于标识一个取样器；服务器名称或IP：HTTP请求发送的目标服务器名称或IP地址；端口号：目标服务器的端口号；方法：发送HTTP请求的方法；Content encoding：内容的编码方式 ；路径：目标URL路径（ (16条消息) 简单介绍url的格式 协议、端口、路径、参数（params）、查询（query）、片段（frag）、url编码_luluoluoa的博客-CSDN博客_url参数格式）

监听器

  • 动态监控JMeter测试执行的结果，实时查看测试结果 。包括结果树和聚合报告。
    • 结果树：可以看到每个请求的结果，红色出错、绿色通过 ，Thread Name ：线程组名称；Sample Start: 启动开始时间；Load time ：加载时长；Latency ：等待时长；Size in bytes ：发送的数据总大小；Headers size in bytes ：发送数据的其余部分大小；Sample Count ：发送统计；Error Count ：交互错误统计；Response code ：返回码；Response message ：返回信息；Response headers ：返回的头部信息
    • 聚合报告： lable ：对应每一个 http 请求，显示的是 http 请求的 Name ，如百度 http 请求 name 为 baidu；#Samples ：表示这一次的测试中一共发出了多少请求；Average ：平均响应时间，指的是所有的请求的平均响应时间，默认的情况下是单个请求的平均响应时间，但当使用了 “ 事务控制器 ”时，则以事物为单位显示平均响应时间；Median ：中位数，也就是 50% 用户的响应时间；90%Line ： 90% 用户的响应时间；Min ：最小响应时间；Max ：最大的响应时；Error% ：本次测试中出现错误的请求的数量 / 请求的总数；Throughput ：吞吐量，默认情况下表示每秒完成的请求数；Recived KB/Sec ：每秒从服务器端接收到的数据量，以 kb 为计算的单位

断言

  • assert，用来判断响应的数据是否符合预期 。包含响应断言，json断言（采用json数据格式），xml断言

定时器

  • 定义请求之间的延迟间隔，就是隔多久请求一次

QTP

  • quick test profession，用于回归测试。需要录制脚本，可以录制以后在代码化脚本修改
